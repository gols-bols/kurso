# План развития серверной части ИС управления заявками (Laravel)

## 1. Назначение документа

Данный файл фиксирует текущее состояние проекта, целевую архитектуру и этапы развития информационной системы управления заявками для ГБПОУ МО «СПК». План приведен в актуализированном виде: он не только описывает желаемую полную конфигурацию, но и отделяет уже реализованный функционал от функций, запланированных на следующие этапы разработки.

Такой подход нужен для двух задач:
- проект выглядит согласованным при показе и защите;
- читатель сразу понимает, что в репозитории уже работает, а что является перспективой расширения.

---

## 2. Краткое описание предметной области

В учебной организации регулярно возникают внутренние обращения, связанные с:
- технической поддержкой рабочих мест;
- проблемами с печатной техникой;
- настройкой сетевого доступа;
- сопровождением программного обеспечения;
- организационно-хозяйственными запросами.

Если подобные обращения обрабатываются устно, через мессенджеры или неформальные записи, возникают типовые проблемы:
- обращения теряются;
- невозможно отследить исполнителя;
- отсутствует единая история изменений;
- руководство не видит общую картину загрузки;
- пользователь не понимает, на каком этапе находится его запрос.

Создаваемая система должна централизовать прием заявок, обеспечить контроль статусов, упростить учет и заложить базу для дальнейшей автоматизации.

---

## 3. Цели проекта

### 3.1 Главная цель

Разработать серверную часть и базовый web-интерфейс системы управления заявками, обеспечивающие регистрацию, хранение и отображение обращений пользователей в единой среде.

### 3.2 Практические цели

- обеспечить стабильную авторизацию пользователей;
- реализовать создание и просмотр заявок;
- подготовить проект к развертыванию на обычном компьютере с PHP, Composer и MySQL;
- оформить кодовую базу и документацию в понятном и презентабельном виде;
- заложить архитектурную основу для расширения до полноценной helpdesk-системы.

---

## 4. Текущее состояние проекта

На момент актуализации плана в проекте уже реализован рабочий минимальный web-сценарий.

### 4.1 Что уже есть в кодовой базе

- Laravel 11 как базовый framework;
- маршруты web-интерфейса;
- контроллер авторизации;
- контроллер для просмотра и создания заявок;
- миграции для таблиц `users`, `tickets`, `sessions`;
- модель `Ticket`;
- модель `User` для стандартной Laravel-аутентификации;
- Blade-шаблоны для входа, просмотра списка и создания заявки;
- базовая серверная валидация входных данных;
- привязка заявок к авторизованному пользователю.

### 4.2 Что уже работает функционально

- вход пользователя по email и паролю;
- выход из системы;
- переход к списку заявок;
- просмотр всех заявок;
- создание новой заявки;
- автоматическая установка статуса `open`;
- сохранение приоритета и автора заявки;
- защита маршрутов через middleware `auth` и `guest`.

### 4.3 Что пока не реализовано

- REST API и префикс `/api/v1`;
- регистрация пользователей через интерфейс;
- роли с различными правами на уровне политик;
- назначение исполнителя;
- изменение статуса отдельной командой;
- комментарии к заявкам;
- загрузка вложений;
- отчеты и аналитика;
- feature-тесты и OpenAPI/Postman-описание.

Именно поэтому в дальнейшем плане нужно различать реализованный минимум и целевую расширенную версию.

## 5. Роли пользователей

На уровне предметной области система ориентирована на три базовые роли.

### 5.1 Пользователь

Пользователь формирует обращение, описывает проблему и отслеживает ее состояние. Это сотрудник, преподаватель или иной участник внутреннего процесса, которому требуется помощь.

### 5.2 Исполнитель или системный администратор

Исполнитель получает заявки в работу, анализирует проблему, выполняет необходимые действия и переводит обращение между стадиями жизненного цикла.

### 5.3 Руководитель или диспетчер

Руководитель контролирует сроки, нагрузку, распределение заявок и может оценивать эффективность обработки обращений.

### 5.4 Реализация в текущей версии

В коде роль хранится в таблице пользователей, но детализированное разграничение прав пока не внедрено. На текущем этапе это приемлемо, потому что проект демонстрирует базовую инфраструктуру, а ролевая модель отнесена к следующему этапу развития.

## 6. Текущая структура данных

### 6.1 Таблица `users`

Текущая таблица пользователей содержит:
- идентификатор;
- имя;
- email;
- пароль;
- роль;
- remember token;
- временные метки создания и обновления.

### 6.2 Таблица `tickets`

Текущая таблица заявок содержит:
- идентификатор;
- заголовок;
- описание;
- приоритет (`low`, `normal`, `high`);
- статус (`open`, `in_progress`, `resolved`, `closed`);
- ссылку на автора через поле `created_by`;
- временные метки.

### 6.3 Таблица `sessions`

Используется стандартная таблица Laravel для хранения пользовательских сессий. Это делает работу авторизации более предсказуемой и упрощает разворачивание проекта на новом окружении.

## 8. Архитектурный подход

### 8.1 Реализованный вариант

Сейчас проект следует классическому MVC-подходу Laravel:
- `routes/web.php` содержит маршруты;
- контроллеры принимают запросы и управляют переходами;
- модели взаимодействуют с таблицами;
- Blade-шаблоны отвечают за отображение;
- middleware защищают приватные разделы.

Такой вариант подходит для учебного проекта и демонстрации начального работающего продукта.

### 8.2 Целевой вариант

По мере развития проект должен перейти к более выраженной многослойной структуре:
- `Controllers` для приема запросов;
- `FormRequest` для валидации;
- `Services` для бизнес-логики;
- `Policies` для авторизации;
- `Resources` для унификации JSON-ответов;
- `Events` и `Listeners` для реакций на критические действия;
- `Feature tests` для проверки пользовательских сценариев.

## 9. Маршруты и пользовательские сценарии

### 9.1 Уже реализованные web-маршруты

- `GET /login` — форма входа;
- `POST /login` — попытка входа;
- `POST /logout` — выход;
- `GET /tickets` — список заявок;
- `GET /tickets/create` — форма создания;
- `POST /tickets` — сохранение новой заявки.

### 9.2 Реализованные сценарии

1. Неавторизованный пользователь открывает форму входа.
2. После успешной аутентификации пользователь попадает в раздел заявок.
3. Пользователь просматривает ранее созданные записи.
4. Пользователь открывает форму новой заявки.
5. После валидации обращение сохраняется в базе данных.

### 9.3 Запланированные сценарии

- редактирование собственной заявки;
- фильтрация списка;
- изменение статуса;
- добавление комментария;
- просмотр подробной карточки;
- прикрепление файлов;
- раздел отчетов.

## 10. Требования к интерфейсу отображения

Так как проект должен выглядеть цивильно и быть готовым к демонстрации, даже минимальный интерфейс обязан отвечать нескольким условиям:

- понятные заголовки страниц;
- единый визуальный стиль;
- читаемые формы и сообщения об ошибках;
- отображение пустых состояний;
- наличие навигации между ключевыми разделами;
- корректная работа на настольных и мобильных экранах.

В текущей актуализации проекта это требование учтено: интерфейс приведен к единой стилистике, формы и списки стали более презентабельными, а шаблоны получили общую основу.

## 11. Требования к безопасности

Для учебного проекта обязательными считаются следующие меры:
- использование хешированных паролей;
- защита CSRF для web-форм;
- серверная валидация входных данных;
- защита приватных маршрутов через middleware;
- ограничение массового присваивания через `$fillable`;
- хранение конфиденциальных настроек в `.env`.

На следующем этапе желательно добавить:
- rate limit на вход;
- более явное логирование критических действий;
- политики доступа;
- аудит изменения статусов;
- резервирование и мониторинг.
