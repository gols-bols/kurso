# ИС управления заявками для системного администратора (Laravel + MySQL)

Проект реализует серверную часть и web-интерфейс системы управления заявками для ГБПОУ МО СПК на базе Laravel.

## Введение
Цель проекта — централизованная обработка заявок пользователей, безопасное хранение данных и разграничение прав доступа (администратор, менеджер, пользователь).

## Основная часть
Реализация выполнена в структуре Laravel-приложения:
- маршруты: `routes/web.php`;
- контроллеры: `app/Http/Controllers/AuthController.php`, `app/Http/Controllers/TicketController.php`;
- модели: `app/Models/Ticket.php`;
- шаблоны интерфейса: `resources/views/...`;
- точка входа: `public/index.php`.

Текущий функционал:
- вход/выход пользователей;
- дашборд со сводкой по статусам, корпусам, приоритетам и исполнителям;
- журнал заявок с фильтрами по статусу, приоритету и корпусу;
- карточка заявки с полной информацией и историей комментариев;
- создание заявок только пользователем-заявителем;
- обработка заявок по корпусам;
- назначение исполнителя администратором;
- изменение рабочих статусов менеджером корпуса и администратором;
- комментарии и автоматические системные записи при изменении статуса;
- REST API v1 с Bearer-токеном для мобильных и внешних клиентов.

## БД
Используется **MySQL**.

### Подключение к БД
В `.env` должны быть параметры:
- `DB_CONNECTION=mysql`
- `DB_HOST=127.0.0.1`
- `DB_PORT=3306`
- `DB_DATABASE=spk_tickets`
- `DB_USERNAME=spk_user`
- `DB_PASSWORD=secret`

Пример есть в `.env.example`.

### Структура БД (миграции)
- `users` — пользователи системы (`name`, `email`, `password`, `role`, `campus`, `api_token`, timestamps).
- `tickets` — заявки (`title`, `description`, `priority`, `status`, `requester_name`, `campus`, `room`, `created_by`, `assignee_id`, timestamps).
- `ticket_comments` — история обращения (`ticket_id`, `user_id`, `type`, `body`, timestamps).

Связи:
- `tickets.created_by -> users.id`
- `tickets.assignee_id -> users.id`
- `ticket_comments.ticket_id -> tickets.id`

## Сайт
Страницы:
- `/login` — авторизация;
- `/dashboard` — аналитическая сводка по заявкам;
- `/tickets` — журнал заявок с фильтрами и ролевым доступом;
- `/tickets/create` — форма создания заявки заявителем;
- `/tickets/{ticket}` — карточка заявки с комментариями;
- `/tickets/{ticket}/edit` — обработка заявки сотрудниками сопровождения.

## REST API
Базовый префикс: `/api/v1`.

Основные методы:
- `POST /login` — получение Bearer-токена;
- `POST /logout` — отзыв токена;
- `GET /tickets` — список заявок с фильтрами;
- `POST /tickets` — создание заявки (роль `user`);
- `GET /tickets/{id}` — карточка заявки с комментариями;
- `PUT /tickets/{id}` — обновление заявки;
- `POST /tickets/{id}/comments` — добавление комментария.

Коллекция Postman: `docs/SPK_Helpdesk_API.postman_collection.json`.

## Приложение Б
Пошаговый запуск:
1. `composer install`
2. `cp .env.example .env`
3. Создать БД MySQL `spk_tickets`
4. `php artisan key:generate`
5. `php artisan migrate --seed`
6. `php artisan serve`

### Тестовые данные
После `php artisan migrate --seed` в системе будут созданы тестовые пользователи:
- `admin@spk.local` / `password`
- `manager-main@spk.local` / `password`
- `manager-k2@spk.local` / `password`
- `user@spk.local` / `password`

Логика ролей:
- `user` — создает заявки и видит только свои обращения;
- `manager` — видит заявки только своего корпуса и меняет рабочие статусы;
- `admin` — видит все корпуса, редактирует все поля и назначает исполнителей.

Также будут добавлены несколько демонстрационных заявок по разным корпусам для проверки интерфейса.
